# 报告 4：从愿景到规划、任务、里程碑的落地方案（与 ai-lib-plans 衔接）

> **性质**：说明如何把报告 1～3 的结论**转写**为可执行工件，**不替代**现 task YAML 的 id。  
> **同步规则**：`ai-lib-plans` 中 [GOV-002] 要求任务与 `MEMORY` 不打架；大叙事变更先审阅再进 `active/projects/*`。

---

## 1. 分层结构（推荐）

| 层 | 存放位置 | 内容 |
|----|----------|------|
| **愿景/叙事** | 仅内审、或经删减进公开 `README` / `ailib.info` | 不绑定具体 task id，避免与闸门冲突。 |
| **阶段路线图** | `ai-lib-plans/active/projects/_meta/` 或单页 `ROADMAP.md`（若已有则扩展） | 按季度/半年，**显式**标注 **Pre-PT-073** / **Post-PT-073**。 |
| **项目** | `active/projects/<name>/` | 如 `ailib-wasm-test`、`ai-protocol`、未来 `ailib-distribution`（示例名）。 |
| **任务** | `active/projects/<name>/tasks/*.yaml` | 可审计、含 executor 字段 [task-executor-terminal]。 |
| **规则** | `ai-lib-constitution/rules/**` | 只收纳**长期稳定**的边界（如 A/B/C 带定义的最终版）。 |

---

## 2. 与当前闸门（PT-073）的关系

- **所有「新产品名称」的代码任务**，在 YAML 中增加字段建议：  
  - `release_dependency: "PT-073"` 或 `none`。  
- **不阻塞 PT-073 的**：`ailib-wasm-test` 的脚本/Docker/文档、benchmark 的基线、协议文档澄清。  
- **应阻塞或排在 PT-073 后的**：**大规模** Contact API 承诺、多运行时「策略路由」GA、企业控制台首版。

---

## 3. 建议的「第一波」可拆任务包（ailib-wasm-test）

以下为**逻辑任务包**，转写为 YAML 时由你方分配 id 与 owner：

1. **可复现构建**：`run` 脚本 / `justfile`、Windows 注意事项；产物路径与 `.gitignore`。  
2. **静态资源路径**：无机器相关绝对路径（原对话已指出）。  
3. **容器**：单阶段或多阶段 `Dockerfile`，文档中 env 与密钥约定。  
4. **双 WASM 线说明**：`README` 中一节「When to use wasm-browser vs ai-lib-wasm (WASI)」。  
5. **v0.2.0 tag**：与 `milestone: v0.2.0` 对齐的 changelog。

---

## 4. 建议的「第二波」任务包（产品化前奏，轻量）

1. **Pack 合同草拟**（在 `ai-protocol` 或新 `docs/`）：仅 JSON Schema 扩展 + 1 个示例，**不**上商店。  
2. **P 层示例**（`ai-lib-rust` 或独立 repo）：**最小**成本路由示例，明确标注 **not production SLA**。  
3. **constitution 备案**：本目录报告定稿后，**抽取** 3～5 条为 `rules/business` 或 `MEMORY` 节，删除重复长文。

---

## 5. 与「plans 同步」规则的关系

若你在 `ai-lib-plans` 中修改了路线图或 MEMORY：

- 按工作区规则执行 `sync_ai_lib_plans.sh` 的 pre-plan / post-doc-change（在可用环境中）。  
- 若脚本路径为 Linux 占位符，在 Windows 上可记录等价手动步骤，避免「文档写了从未执行」。
