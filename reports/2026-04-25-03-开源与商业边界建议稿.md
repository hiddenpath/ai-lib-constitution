# 报告 3：开源与商业边界（修订建议稿，拟进入 constitution / 规则前审阅）

> **性质**：与《ai-lib开源分界产品规划和商业模式》中 Open Core 讨论对齐，但**以事实**「manifest schema 已在 ai-protocol 公开」**为起点**，避免「藏不住」与治理来回打架。  
> **与现有原则对齐**：[ARCH-001] 协议优先、[DOC-002] 内部与公开边界、[GOV-001] 远程与仓库政策。

---

## 1. 目标

- 开源侧：**可参与、可 fork、可审计**，吸引多运行时实现与 community manifests。  
- 商业侧：在**不收回 schema** 的前提下，获得可持续收入（企业、规模、合规模块、受信分发、SLA）。  
- 避免：公开仓库里出现**半吊子闭源功能**或含糊授权，导致社区信任崩盘。

---

## 2. 建议边界（三带模型）

### 2.1 A 带 — 永久开源、标准与参考实现

**建议保持 Apache-2.0（与现有一致则不改）**：

- `ai-protocol`：**全部** schema、示例 manifest、合规测试夹具、文档。  
- 各语言 **Core（E）** 与**协议加载/执行**路径、合规测试。  
- `ailib-wasm-test`、demo、教程、**不含** 付费密钥或仅内部 endpoint 的 `run` 配置。  
- `ai-lib-benchmark` 的**方法与基线**（原始数据脱敏、可复现 — [BENCH-001] 纪律）。

**原则**：**标准与「能互操作的最小实现」在 A 带**；这是生态的**共同语言**。

### 2.2 B 带 — 开源但明确「非 SLA / 自承载」

- **Contact（P）** 的参考实现、示例策略、示例路由 —— 可开源以教育市场，但 README 中声明：**非企业 SLA**、**不保证**与某云厂商的最新限额一致。  
- spiderswitch：保持**执行面**与 MCP 能力；**不**在开源仓库承诺「商业级多租户路由大脑」全功能。

**原则**：B 带降低采用门槛，但**不透支企业承诺**。

### 2.3 C 带 — 可闭源 / 双授权 / 仅二进制的商业模块

| 内容示例 | 理由 |
|----------|------|
| 受信 Registry（签名校验、撤包、多租户） | 运营与安全责任 |
| 企业控制台、审计报表、SSO 连接器 | 集成与支持成本 |
| 垂直行业「认证 Pack」内容（在合同允许范围内） | 内容与服务 |
| 7×24、专属 CVE 响应、现场部署 | 服务而非代码 |

**原则**：C 带卖的是**信任、责任范围与集成深度**，不是「没有 schema 就不可用」的勒索式闭源。

---

## 3. 与原文档「藏 schema / 只闭源解析器」说法的修正

- **已过时**：「manifest 解析器必须闭源才能保密」。  
- **替换为**：  
  - **公共解析器**继续开源（与多运行时一致）。  
  - **企业增强**：签名链验证、多 registry 回退、**策略生效证明**、与 Fleet 管理集成 —— 可 C 带。  
  - **受信目录**：metadata、评分、弃用、法律分类 —— 典型**运营资产**，不必开源也能成立。

---

## 4. License 与社区沟通（短条款示例，非最终法条）

- 公开放：Apache-2.0 为默认。  
- 若某模块使用 **AGPL** 或 **BSL/变更版**：须**单仓声明**、在根 README 与 `ai-lib-constitution` 备案，避免「混源」被误认为全 Apache。  
- 商业二开客户：在合同中明确 **不暗示** ailib-official 背书其 fork 的安全性。

（具体法条与 SPDX 头由法务/项目所有者定稿。）

---

## 5. 可写入 MEMORY 的「一句决策」草案

> 在 **manifest schema 全公开** 的前提下，商业边界定义为：**受信分发、企业治理与集成、SLA 与合规模块（C 带）**；**协议互操作与核心执行（A 带）** 继续开源。策略性路由与复杂弹性属于 **P 或上层产品**，与 spiderswitch 执行面边界不混淆。

（完）
