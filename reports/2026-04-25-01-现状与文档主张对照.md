## 2. 已验证的现状态（以 MEMORY / 任务为准）

### 2.1 组织与工程现实

- **规范与实现分离**：`ai-protocol` 管 schema + manifests；`ai-lib-*` 四语言 + `ai-lib-rust` 内 `ai-lib-core` / `ai-lib-contact` / `ai-lib-wasm` 分工明确。  
- **v1.0 级闸门**：`MEMORY` 中 **PT-073** 为当前主开放门（v1.0 RC），与 E/P 分离、WASM 合规子集、多运行时 CI 切片等相关；**未完成的发布纪律会约束一切「新产品」节奏**。  
- **E/P 分离（`MEMORY` 中已决）**  
  - **E（core）**：确定性能力执行、微重试、协议驱动。  
  - **P（contact）**：路由、缓存、批处理、复杂弹性、**策略引擎**、遥测、护栏等。  
  - **WASM 定位**：为控制体积与可移植性，**以 core-only 路径为主**；策略层不强行塞进 E。

### 2.2 ailib-wasm-test（`project-overview.yaml`）

- **版本/里程碑**：`current_version: v0.1.1`，`milestone: v0.2.0`。  
- **双 WASM 线并存**（与文中仅强调「浏览器 demo」相比，**工程上更细**）：  
  1. `wasm-browser`（wasm-bindgen）— 浏览器聊天 demo；  
  2. `ai-lib-wasm`（C ABI / WASI）— `ai-lib-rust` 内，wasmtime 等侧。  
- **WASM 加固任务 WASM-001～003**：在计划记录中为 **completed**（ABI、内存、状态迁移等）。  
- **结论**：将该项目称为「能力分发平台最小闭环的合适载体」是合理的，但**严谨表述**应是：**浏览器端协议执行闭环 + 薄代理** 的参考实现，而不是已经包含「平台级 manifest 分发/计费」的完整产品。

### 2.3 spiderswitch（`MEMORY` 中 Runtime Routing Boundary）

- 明确定位：**能力路由执行面**，**不是**业务策略引擎。  
- **策略**（成本、租户规则、智能 fallback 链的**业务含义**）留在**上层应用**或 **Contact 层**，而非把 spiderswitch 直接升格为「AI 能力路由器产品内核」。

这对原文档中多次把 **spiderswitch + WASM** 直接等同于「可分发的大一统 Router 引擎」的叙述，构成**直接架构约束**。

## 3. 文档主张与现状的对照结论

| 文档主张 | 与现状关系 | 说明 |
|----------|------------|------|
| 协议进 WASM、薄代理 | **强一致** | 与 `ailib-wasm-test` 设计、ARCH-001/003 一致。 |
| 「路由与分发能力」是差异化 | **方向一致，需拆层** | 分发/策略应在 **P 或独立产品** 定义；避免写进 **E** 或错误绑定 **spiderswitch 职责**。 |
| Capability Pack / Marketplace | **长期可行，当前缺少独立交付物** | 需以 `ai-protocol` 公开 schema 为合同，另立 **pack 规范、签名、发布渠道** 任务，避免与 PT-073 抢资源。 |
| Distributed AI Mesh / P2P | **技术可行，商业与工程风险高** | `MEMORY` 无现成 P2P 基座；应标为**探索/长期**，不可与「最小闭环」并行作为 P0。 |
| Open Core + 闭源 Registry/企业功能 | **模式可行，需与 License/社区预期文书一致** | 见报告 2。 |
| 先把 wasm-test 做到「可上线演示」 | **与 `milestone v0.2.0` 一致** | 工程项（可移植路径、一键脚本、容器、产物忽略策略等）应进 `ailib-wasm-test` 任务，而非只停留在口头。 |

## 4. 对报告 1 的结论（给审阅人）

1. 原文档的**市场叙事**与 **ai-lib 的协议+多运行时** 故事对齐；但 **技术落点** 必须与 **E/P 分离**、**spiderswitch 边界** 两大治理决议一致，否则后续产品文档会与 `MEMORY` 冲突。  
2. **当前最硬的工程事实**是：PT-073 与 v1.0 RC 纪律；任何商业产品线规划都应在任务层面 **显式标注依赖**（例如「Post-PT-073」），避免组织层面过度承诺。  
3. **ailib-wasm-test** 作为「车和路的起点」合理，但应写成：**v0.2.x = 可复现的公开演示与打包形态**，再谈 Pack/Router/Console。

---

## 5. 建议写入 constitution / plans 的「最小摘要句」（供改稿时选用）

- **产品叙事**：差异化在「协议可移植执行 + 可选策略与分发层」，**不是**在单一 repo 里堆全部能力。  
- **spiderswitch**：保持执行与运行时切换；**不**作为唯一「智能商业路由」品牌内核，除非单独立项并改治理表述。  
- **商业边界**：公开 schema 与基础加载器；付费侧重 **信任、规模、合规模块与运营**，见报告 2。
