# 报告 2：原商业与产品构想的可行性评估与风险矩阵

> **性质**：内部审阅稿；用于判断哪些建议**现在可做**、**条件可做**、**应暂缓**。

---

## 1. 评估维度

对原文档中每条「产品/变现」建议，用以下维度打分（高/中/低）：

- **与现有资产贴合度**：是否直接复用 `ai-protocol`、`ai-lib-*`、`ailib-wasm-test`、spiderswitch。  
- **架构一致性**：是否违反 E/P 分离、spiderswitch 角色、ARCH-001。  
- **依赖闸门**：是否被 PT-073、合规矩阵、WASM 策略阻塞。  
- **验证成本**：从想法到**可证伪** Demo 的周期。  
- **单维护者/小团队可承受性**：文档未假设团队规模，需保守。

---

## 2. 分项评估

### 2.1 ailib-wasm-test → 「基本可用 / 可对外演示」v0.2.x

| 维度 | 结论 |
|------|------|
| 资产贴合度 | **高** — 仓库与 `project-overview` 已描述完整栈。 |
| 架构一致性 | **高** — 薄代理 + 浏览器 WASM 与 core 职责一致。 |
| 依赖闸门 | **中** — 不等待 PT-073 也可推进演示工程化；与核心发布节奏可并行但需**明确版本线**。 |
| 验证成本 | **低** — 可复现构建、Docker、README 即验证。 |
| 可承受性 | **高** |

**风险**：双 WASM 线（browser vs WASI）导致文档/品牌混淆 —— 对外叙事需区分 **「浏览器协议 Demo」** 与 **「WASI 侧 ai-lib-wasm」**，避免用户以为一个 npm 包解决全部场景。

**判定**：**强烈建议作为 P0**，且作为后续一切「能力分发故事」的**硬证据**。

---

### 2.2「AI-Capability Router」作为商业单品

| 维度 | 结论 |
|------|------|
| 资产贴合度 | **中** — 有 core 的 build/parse，无现成**统一商业路由**仓库。 |
| 架构一致性 | **需设计** — 策略路由属 **P（ai-lib-contact）** 或独立服务；**不要**用 spiderswitch 名义承载全部「智能」以免违反现有边界定义。 |
| 依赖闸门 | **高** — 与 PT-073、E/P 文档、Contact API 稳定度相关。 |
| 验证成本 | **中高** — 要定义可测的「路由」语义 + 跨运行时一致。 |
| 可承受性 | **中** |

**判定**：**可行，但应降级为「Contact 或独立 crate / 二进制的 Phase 2」**；MVP 不妨叫 **「策略示例 + 可插拔 P 层」**，避免与 OpenRouter 做不严谨对标。

---

### 2.3 Capability Pack + Marketplace

| 维度 | 结论 |
|------|------|
| 资产贴合度 | **中高** — schema 在 `ai-protocol`；**缺** pack 格式、签名、索引协议。 |
| 架构一致性 | **高** — 纯配置 + 可选扩展，可遵守 ARCH-001。 |
| 依赖闸门 | **中** — 不阻塞 PT-073，但 **Marketplace 运营**与**合规**工作量大。 |
| 验证成本 | **中** — 可先做「GitHub Release + 校验脚本」伪 Marketplace。 |
| 可承受性 | **中低**（运营与法务） |

**判定**：**商业上合理（高毛利叙事成立）**；**工程上**应拆为：**公开 pack 合同草案 → 一个权威示例 pack → 再谈商店**。避免一上来做完整「应用商店」。

---

### 2.4 Enterprise Console（审计、SSO、策略推送）

| 维度 | 结论 |
|------|------|
| 资产贴合度 | **低** — 无现成控制面前端仓库。 |
| 架构一致性 | **中** — 可挂在 P/策略与日志管线。 |
| 依赖闸门 | **高** — 企业采购、安全、SLA 与当前开源成熟度可能不匹配。 |
| 验证成本 | **高** |
| 可承受性 | **低**（小团队） |

**判定**：**建议标为 12～18 个月级或「有首单企业需求后再立项」**；可先做**设计文档 + API 草图**占位，不扩散实现。

---

### 2.5 Distributed AI Mesh / P2P 分发

| 维度 | 结论 |
|------|------|
| 资产贴合度 | **低** — MEMORY 未体现 P2P 基线。 |
| 架构一致性 | **未定** — 需全新信任模型。 |
| 依赖闸门 | **极高** |
| 验证成本 | **极高** |
| 可承受性 | **极低**（短期） |

**判定**：**作为愿景保留，不应进入 12 个月可执行路线图的承诺项**。

---

### 2.6 Benchmark Pro（SaaS）

| 维度 | 结论 |
|------|------|
| 资产贴合度 | **中** — `ai-lib-benchmark` 已独立。 |
| 架构一致性 | **高** |
| 依赖闸门 | **中** — 需可重复工件与 BENCH-001 纪律。 |
| 验证成本 | **中** — 可先「手动报告 + 固定 baseline」再 SaaS 化。 |
| 可承受性 | **中** |

**判定**：**可行**，适合作为**开源信誉建设**后自然延伸，不宜早于「可复现公开基准」口碑。

---

### 2.7 托管「透明协议」Proxy（类 OpenRouter）

| 维度 | 结论 |
|------|------|
| 资产贴合度 | **中** — 有 mock、有代理模式，**无**现成多租户公网 SRE 体系。 |
| 架构一致性 | **中** — 与「客户端 WASM」故事部分重叠，易变成又一中心化点。 |
| 依赖闸门 | **高** — 安全、滥用、账密、地区合规。 |
| 验证成本 | **高** |
| 可承受性 | **低**（早期） |

**判定**：**仅当**有明确**商业模式与运维**时再立项；**不应**与「去中心/WASM 差异化」同页混讲而不解释定位。

---

## 3. 总风险矩阵（简表）

| 建议项 | 可行性 | 主要风险 | 建议阶段 |
|--------|--------|----------|----------|
| wasm-test 工程化 v0.2 | **高** | 双 WASM 叙事混乱 | 立即 |
| Pack 合同 + 1 个示例 | **中高** | 法律/ToS、维护成本 | 0～6 月 |
| P 层路由 SDK / 参考实现 | **中** | 与 spiderswitch/PT-073 边界 | 6～12 月视闸门 |
| Enterprise Console | **中低** | 销售周期、安全审计 | 12 月+ |
| Mesh / P2P | **低** | 技术+信任 | 愿景/研究 |
| 公网 Multi-tenant Proxy | **中低** | 运维与滥用 | 有团队与首单后 |

---

## 4. 结论（给审阅人）

原文档的**方向**在开源基础设施领域有合理性，但若原样写进对外的「一年变成基础设施公司」类时间表，**与 PT-073、E/P 治理、小团队现实** 不相容。  

**可落地的序曲**应收缩为：

1. **硬**：ailib-wasm-test v0.2 可复现、可容器、文档清晰。  
2. **软**：公开「策略在 P/上层」的技术叙事，不重塑 spiderswitch 定义。  
3. **商**：在 schema 已公开的前提下，**商业墙 = 运营 + 企业特性 + 签名分发**，而非「藏 schema」。

（完）
